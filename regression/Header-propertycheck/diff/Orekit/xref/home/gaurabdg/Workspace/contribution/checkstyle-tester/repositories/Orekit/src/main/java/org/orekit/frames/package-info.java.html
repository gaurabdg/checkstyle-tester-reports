17  /**
18   *
19   * This package provides classes to handle frames and transforms between them.
20   *
21   * <p>
22   * The {@link org.orekit.frames.Transform} class represents a full transform:
23   * combined rotation and translation, and their first time derivatives to handle kinematics.
24   * </p>
25   *
26   * <p>
27   * Each {@link org.orekit.frames.Frame} is defined by a transform linking it to
28   * another one, called its parent frame. The only exception is the root frame which has
29   * no parent. This implies that all frames are naturally organized as a tree with a single
30   * root. The predefined GCRF inertial frame was arbitrary chosen as the root for every tree.
31   * </p>
32   *
33   * <p>
34   * The {@link org.orekit.frames.FramesFactory} class implements several predefined reference
35   * frames. One set correspond to the frames from the various IERS conventions (ITRF and others).
36   * Other frames not belonging to the previous set are the EME2000 frame that was used prior
37   * to GCRF and which is linked to GCRF by a simple bias rotation, the MOD (Mean Of Date) frame
38   * which involves the IAU 1976 precession model, the TOD (True Of Date) frame which involves
39   * the IAU 1980 nutation model, the GTOD (Greenwich True Of Date) which involves the IAU 1982
40   * Greenwich sidereal time model and the Veis 1950 frame which involves a Veis modified sidereal
41   * time model.
42   * </p>
43   *
44   * <p>
45   * Some other frames are predefined outside of this package, in the {@link
46   * org.orekit.bodies.CelestialBodyFactory} class. They correspond to the Sun, Moon, planets, solar
47   * system barycenter and Earth-Moon barycenter. For convenience, the very important solar system
48   * barycenter frame, which is the ICRF, can also be retrieved from the factory in this package
49   * even if it is really implemented in the bodies package.
50   * </p>
51   *
52   * <p>
53   * The frames can be time dependent (for example the ITRF frame depends on time due to
54   * precession/nutation, Earth rotation and pole motion). In order to get a transform from one
55   * frame to another one, the date must be specified, and {@link
56   * org.orekit.frames.TransformProvider#getTransform(AbsoluteDate)} is called under the hood.
57   * If a user wants to implement his own date synchronized frame, he has to  implement his
58   * own {@link org.orekit.frames.TransformProvider} class and provide it to the frame constructor.
59   * </p>
60   *
61   * @author Luc Maisonobe
62   * @author Fabien Maussion
63   * @author Véronique Pommier-Maurussane
64   * @author Pascal Parraud
65   *
66   */
67  package org.orekit.frames;
