17  /**
18   *
19   * This package provides classes to represent orbits.
20   *
21   * <p>
22   * It builds the base of all the other space mechanics tools.
23   * It provides an abstract class, {@link org.orekit.orbits.Orbit}, extended
24   * in four different ways corresponding to the different possible representations
25   * of orbital parameters. As of version 3.0, {@link org.orekit.orbits.KeplerianOrbit
26   * keplerian}, {@link org.orekit.orbits.CircularOrbit circular}, {@link
27   * org.orekit.orbits.EquinoctialOrbit equinoctial} and {@link
28   * org.orekit.orbits.CartesianOrbit cartesian} representations are supported.
29   * </p>
30   *
31   * <h3>Design History</h3>
32   * <p>
33   * Early designs for the orbit package were much more complex than the current design.
34   * Looking back at these designs, they tried to do far too much in a single class and
35   * resulted in huge systems which were both difficult to understand, difficult to
36   * use, difficult to maintain and difficult to reuse. They mixed several different notions:
37   * <ul>
38   *   <li>representation (keplerian, cartesian ...)</li>
39   *   <li>kinematics (parameters) and dynamics (propagation)</li>
40   *   <li>physical models (complete or simplified force models, often implicitly assumed with no real reference)</li>
41   *   <li>filtering (osculating and centered or mean parameters, related to some models)</li>
42   * </ul>
43   * <p> They also often forgot all frames issues.
44   *
45   * <p>
46   * The current design has been reached by progressively removing spurious layers and
47   * setting them apart in dedicated packages. All these notions are now still handled,
48   * but all in different classes that are cleanly linked to each other. Without knowing
49   * it, we have followed Antoine de Saint Exup&eacute;ry's saying:
50   * </p>
51   * <p>
52   * <i>It seems that perfection is reached not when there is nothing left to add, but
53   * when there is nothing left to take away</i>.
54   * </p>
55   *
56   * <p>
57   * The current design is not perfect, of course, but it is easy to understand, easy to use,
58   * easy to maintain and reusable.
59   * </p>
60   *
61   * <h3>Current state versus evolving state</h3>
62   * <p>
63   * From the early design, the various orbit classes retained only the kinematical
64   * notions at a single time. They basically represent the current state, and
65   * serve as data holder. Most of methods provided by these orbits classes are
66   * getters. Some of them (the non-ambiguous ones that must be always available) are
67   * defined in the top {@link org.orekit.orbits.Orbit Orbit} abstract class
68   * ({@link org.orekit.orbits.Orbit#getA() Orbit.getA()}, {@link
69   * org.orekit.orbits.Orbit#getDate() Orbit.getDate()}, {@link
70   * org.orekit.orbits.Orbit#getPVCoordinates() Orbit.getPVCoordinates()} ...). The
71   * more specific ones depending on the type of representation are only defined
72   * in the corresponding class ({@link
73   * org.orekit.orbits.CircularOrbit#getCircularEx() CircularOrbit.getCircularEx()}
74   * for example).
75   * </p>
76   *
77   * <p>
78   * It is important to note that some parameters are available even if they seem
79   * out of place with regard to the representation. For example the semi-major axis is
80   * available even in cartesian representation and the position/velocity even in
81   * non-cartesian representation. This choice is a pragmatic one. These parameters
82   * are really used in many places in algorithms, for computation related to
83   * period (setting a convergence threshold or a search interval) or geometry
84   * (computing swath or lighting). A side-effect of this choice is that all orbits
85   * do include a value for μ, the acceleration coefficient of the central body.
86   * This value is only used for the representation of the parameters and for conversion
87   * purposes, it is <em>not</em> always the same as the value used for propagation (but
88   * of course, using different values should be done with care).
89   * </p>
90   *
91   * <p>
92   * Orbits also include a reference to a defining frame. This allows transparent
93   * conversions to other frames without having to externally preserve a mapping
94   * between orbits and their frame: it is already done. As an example, getting
95   * the position and velocity of a satellite given by a circular orbit in a ground
96   * station frame is simply a matter of calling <tt>orbit.getPVCoordinates(stationFrame)</tt>,
97   * regardless of the frame in which the orbit is defined (GCRF, EME2000, ...).
98   * </p>
99   *
100  * <p>
101  * Since orbits are used everywhere in space dynamics applications and since we
102  * have chosen to restrict them to a simple state holder, all orbit classes are
103  * guaranteed to be immutable. They are small objects and they are shared by
104  * many parts. This change was done in 2006 and tremendously simplified the
105  * library and the users applications that were previously forced to copy orbits
106  * as an application of defensive programming, and that were plagued by difficult
107  * to find bugs when they forgot to copy.
108  * </p>
109  *
110  * <p>
111  * For orbit evolution computation, this package is not sufficient. There is a
112  * need to include notions of dynamics, forces models, propagation algorithms ...
113  * The entry point for this is the {@link org.orekit.propagation.Propagator Propagator}
114  * interface.
115  * </p>
116  *
117  * <h3>Representations Conversions</h3>
118  * <p>
119  * All representations can be converted into all other ones. No error is triggered
120  * if some conversion is ambiguous (like converting a perfectly circular orbit from
121  * cartesian representation to keplerian representation, with an ambiguity on the
122  * perigee argument). This design choice is the result of <strong>many</strong>
123  * different attempts and pragmatic considerations. The rationale is that from a
124  * physical point of view, there is no singularity. The singularity is only introduced
125  * by a choice of <em>representations</em>. Even considering this, it appears that
126  * rather than having a parameter with <em>no</em> realistic value, there is an
127  * <em>infinite</em> possible number of values that all represent the same physical
128  * orbit. Orekit simply does an arbitrary choice, often choosing simply the value 0.
129  * In our example case, we would then get a converted orbit with a 0 perigee argument.
130  * This choice is valid, just as any other choice (π/2, π, whatever ...) would
131  * have been valid, in the sense that it <i>does</i> represent correctly the orbit
132  * and when converted back to the original non-ambiguous representation it does give
133  * the right result.
134  * </p>
135  *
136  * <p>
137  * We therefore consider it is the responsibility of the user to be aware of the correct
138  * definition of the different representations and of the singularities relative to each
139  * one of them. If the user really needs to do some conversion (for example to provide
140  * an orbit as Two-Lines Elements later on, remembering than TLE do use keplerian-like
141  * parameters), then he can do so.
142  * </p>
143  *
144  * <p>
145  * The way conversion is handled in OREKIT is very simple and allows easy and transparent
146  * processing. All four parameters type extend the abstract class {@link org.orekit.orbits.Orbit},
147  * and every propagation algorithm, be it analytical or numerical, use this abstract class as a
148  * parameter, even if internally it relies on a specific representation. As an example, the
149  * {@link org.orekit.propagation.analytical.EcksteinHechlerPropagator Eckstein-Hechler propagator}
150  * is defined in terms of {@link org.orekit.orbits.CircularOrbit circular orbit} only. So
151  * there is an implicit conversion done at propagator initialization time.
152  * </p>
153  *
154  * @author Luc Maisonobe
155  * @author Fabien Maussion
156  * @author Véronique Pommier-Maurussane
157  *
158  */
