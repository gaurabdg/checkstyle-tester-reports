32   <h2><a name="overview">Overview</a></h2>
33   Constraints are used to enforce business rules in a database.
34   By checking all {@link org.apache.hadoop.hbase.client.Put Puts} on a given table, you can enforce very specific data policies.
35   For instance, you can ensure that a certain column family-column qualifier pair always has a value between 1 and 10.
36   Otherwise, the {@link org.apache.hadoop.hbase.client.Put} is rejected and the data integrity is maintained.
37   <p>
38   Constraints are designed to be configurable, so a constraint can be used across different tables, but implement different
39   behavior depending on the specific configuration given to that constraint.
40   <p>
41   By adding a constraint to a table (see <a href="#usage">Example Usage</a>), constraints will automatically enabled.
42   You also then have the option of to disable (just 'turn off') or remove (delete all associated information) all constraints on a table.
43   If you remove all constraints
44   (see {@link org.apache.hadoop.hbase.constraint.Constraints#remove(org.apache.hadoop.hbase.HTableDescriptor)},
45   you must re-add any {@link org.apache.hadoop.hbase.constraint.Constraint} you want on that table.
46   However, if they are just disabled (see {@link org.apache.hadoop.hbase.constraint.Constraints#disable(org.apache.hadoop.hbase.HTableDescriptor)},
47   all you need to do is enable constraints again, and everything will be turned back on as it was configured.
48   Individual constraints can also be individually enabled, disabled or removed without affecting other constraints.
49   <p>
50   By default, constraints are disabled on a table.
51   This means you will not see <i>any</i> slow down on a table if constraints are not enabled.
52   <p>
53  
54   <h2><a name="concurrency">Concurrency and Atomicity</a></h2>
55   Currently, no attempts at enforcing correctness in a multi-threaded scenario when modifying a constraint, via
56   {@link org.apache.hadoop.hbase.constraint.Constraints}, to the the {@link org.apache.hadoop.hbase.HTableDescriptor}.
57   This is particularly important when adding a constraint(s) to the {@link org.apache.hadoop.hbase.HTableDescriptor}
58   as it first retrieves the next priority from a custom value set in the descriptor,
59   adds each constraint (with increasing priority) to the descriptor, and then the next available priority is re-stored
60   back in the {@link org.apache.hadoop.hbase.HTableDescriptor}.
61   <p>
62   Locking is recommended around each of Constraints add methods:
63   {@link org.apache.hadoop.hbase.constraint.Constraints#add(org.apache.hadoop.hbase.HTableDescriptor, Class...)},
64   {@link org.apache.hadoop.hbase.constraint.Constraints#add(org.apache.hadoop.hbase.HTableDescriptor, org.apache.hadoop.hbase.util.Pair...)},
65   and {@link org.apache.hadoop.hbase.constraint.Constraints#add(org.apache.hadoop.hbase.HTableDescriptor, Class, org.apache.hadoop.conf.Configuration)}.
66   Any changes on <i>a single HTableDescriptor</i> should be serialized, either within a single thread or via external mechanisms.
67   <p>
68   Note that having a higher priority means that a constraint will run later; e.g. a constraint with priority 1 will run before a
69   constraint with priority 2.
70   <p>
71   Since Constraints currently are designed to just implement simple checks (e.g. is the value in the right range), there will
72   be no atomicity conflicts.
73   Even if one of the puts finishes the constraint first, the single row will not be corrupted and the 'fastest' write will win;
74   the underlying region takes care of breaking the tie and ensuring that writes get serialized to the table.
75   So yes, this doesn't ensure that we are going to get specific ordering or even a fully consistent view of the underlying data.
76   <p>
77   Each constraint should only use local/instance variables, unless doing more advanced usage. Static variables could cause difficulties
78   when checking concurrent writes to the same region, leading to either highly locked situations (decreasing through-put) or higher probability of errors.
79   However, as long as each constraint just uses local variables, each thread interacting with the constraint will execute correctly and efficiently.

81   <h2><a name="caveats">Caveats</a></h2>
82   In traditional (SQL) databases, Constraints are often used to enforce <a href="http://en.wikipedia.org/wiki/Relational_database#Constraints">referential integrity</a>.
83   However, in HBase, this will likely cause significant overhead and dramatically decrease the number of
84   {@link org.apache.hadoop.hbase.client.Put Puts}/second possible on a table. This is because to check the referential integrity
85   when making a {@link org.apache.hadoop.hbase.client.Put}, one must block on a scan for the 'remote' table, checking for the valid reference.
86   For millions of {@link org.apache.hadoop.hbase.client.Put Puts} a second, this will breakdown very quickly.
87   There are several options around the blocking behavior including, but not limited to:
88   <ul>
89   <li>Create a 'pre-join' table where the keys are already denormalized</li>
90   <li>Designing for 'incorrect' references</li>
91   <li>Using an external enforcement mechanism</li>
92   </ul>
93   There are also several general considerations that must be taken into account, when using Constraints:
94   <ol>
95   <li>All changes made via {@link org.apache.hadoop.hbase.constraint.Constraints} will make modifications to the
96   {@link org.apache.hadoop.hbase.HTableDescriptor} for a given table. As such, the usual renabling of tables should be used for
97   propagating changes to the table. When at all possible, Constraints should be added to the table before the table is created.</li>
98   <li>Constraints are run in the order that they are added to a table. This has implications for what order constraints should
99   be added to a table.</li>
100  <li>Whenever new Constraint jars are added to a region server, those region servers need to go through a rolling restart to
101  make sure that they pick up the new jars and can enable the new constraints.</li>
102  <li>There are certain keys that are reserved for the Configuration namespace:
103  <ul>
104  <li>_ENABLED - used server-side to determine if a constraint should be run</li>
105  <li>_PRIORITY - used server-side to determine what order a constraint should be run</li>
106  </ul>
107  If these items are set, they will be respected in the constraint configuration, but they are taken care of by default in when
108  adding constraints to an {@link org.apache.hadoop.hbase.HTableDescriptor} via the usual method.</li>
109  </ol>
110  <p>
111  Under the hood, constraints are implemented as a Coprocessor (see {@link org.apache.hadoop.hbase.constraint.ConstraintProcessor}
112  if you are interested).

115  <h2><a name="usage">Example usage</a></h2>
116  First, you must define a {@link org.apache.hadoop.hbase.constraint.Constraint}.
117  The best way to do this is to extend {@link org.apache.hadoop.hbase.constraint.BaseConstraint}, which takes care of some of the more
118  mundane details of using a {@link org.apache.hadoop.hbase.constraint.Constraint}.
119  <p>
120  Let's look at one possible implementation of a constraint - an IntegerConstraint(there are also several simple examples in the tests).
121  The IntegerConstraint checks to make sure that the value is a String-encoded <code>int</code>.
122  It is really simple to implement this kind of constraint, the only method needs to be implemented is
123  {@link org.apache.hadoop.hbase.constraint.Constraint#check(org.apache.hadoop.hbase.client.Put)}:
124 
125  <div style="background-color: #cccccc; padding: 2px">
126  <blockquote><pre>
127  public class IntegerConstraint extends BaseConstraint {
128  public void check(Put p) throws ConstraintException {
129 
130  Map&ltbyte[], List&ltKeyValue&gt&gt familyMap = p.getFamilyMap();
131 
132  for (List &ltKeyValue&gt kvs : familyMap.values()) {
133  for (KeyValue kv : kvs) {
134 
135  // just make sure that we can actually pull out an int
136  // this will automatically throw a NumberFormatException if we try to
137  // store something that isn't an Integer.
138 
139  try {
140  Integer.parseInt(new String(kv.getValue()));
141  } catch (NumberFormatException e) {
142  throw new ConstraintException("Value in Put (" + p
143  + ") was not a String-encoded integer", e);
144  } } }
145  </pre></blockquote>
146  </div>
147  <p>
148  Note that all exceptions that you expect to be thrown must be caught and then rethrown as a
149  {@link org.apache.hadoop.hbase.constraint.ConstraintException}. This way, you can be sure that a
150  {@link org.apache.hadoop.hbase.client.Put} fails for an expected reason, rather than for any reason.
151  For example, an {@link java.lang.OutOfMemoryError} is probably indicative of an inherent problem in
152  the {@link org.apache.hadoop.hbase.constraint.Constraint}, rather than a failed {@link org.apache.hadoop.hbase.client.Put}.
153  <p>
154  If an unexpected exception is thrown (for example, any kind of uncaught {@link java.lang.RuntimeException}),
155  constraint-checking will be 'unloaded' from the regionserver where that error occurred.
156  This means no further {@link org.apache.hadoop.hbase.constraint.Constraint Constraints} will be checked on that server
157  until it is reloaded. This is done to ensure the system remains as available as possible.
158  Therefore, be careful when writing your own Constraint.
159  <p>
160  So now that we have a Constraint, we want to add it to a table. It's as easy as:
161 
162  <div style="background-color: #cccccc; padding: 2px">
163  <blockquote><pre>
164  HTableDescriptor desc = new HTableDescriptor(TABLE_NAME);
165  ...
166  Constraints.add(desc, IntegerConstraint.class);
167  </pre></blockquote></div>
168  <p>
169  Once we added the IntegerConstraint, constraints will be enabled on the table (once it is created) and
170  we will always check to make sure that the value is an String-encoded integer.
171  <p>
172  However, suppose we also write our own constraint, <code>MyConstraint.java</code>.
173  First, you need to make sure this class-files are in the classpath (in a jar) on the regionserver where
174  that constraint will be run (this could require a rolling restart on the region server - see <a href="#caveats">Caveats</a> above)
175  <p>
176  Suppose that MyConstraint also uses a Configuration (see {@link org.apache.hadoop.hbase.constraint.Constraint#getConf()}).
177  Then adding MyConstraint looks like this:
178 
179  <div style="background-color: #cccccc; padding: 2px">
180  <blockquote><pre>
181  HTableDescriptor desc = new HTableDescriptor(TABLE_NAME);
182  Configuration conf = new Configuration(false);
183  ...
184  (add values to the conf)
185  (modify the table descriptor)
186  ...
187  Constraints.add(desc, new Pair(MyConstraint.class, conf));
188  </pre></blockquote></div>
189  <p>
190  At this point we added both the IntegerConstraint and MyConstraint to the table, the IntegerConstraint
191  <i>will be run first</i>, followed by MyConstraint.
192  <p>
193  Suppose we realize that the {@link org.apache.hadoop.conf.Configuration} for MyConstraint is actually wrong
194  when it was added to the table. Note, when it is added to the table, it is <i>not</i> added by reference,
195  but is instead copied into the {@link org.apache.hadoop.hbase.HTableDescriptor}.
196  Thus, to change the {@link org.apache.hadoop.conf.Configuration} we are using for MyConstraint, we need to do this:
197 
198  <div style="background-color: #cccccc; padding: 2px">
199  <blockquote><pre>
200  (add/modify the conf)
201  ...
202  Constraints.setConfiguration(desc, MyConstraint.class, conf);
203  </pre></blockquote></div>
204  <p>
205  This will overwrite the previous configuration for MyConstraint, but <i>not</i> change the order of the
206  constraint nor if it is enabled/disabled.
207  <p>
208  Note that the same constraint class can be added multiple times to a table without repercussion.
209  A use case for this is the same constraint working differently based on its configuration.
210 
211  <p>
212  Suppose then we want to disable <i>just</i> MyConstraint. Its as easy as:
213  <div style="background-color: #cccccc">
214  <blockquote><pre>
215  Constraints.disable(desc, MyConstraint.class);
216  </pre></blockquote></div>
217  <p>
218  This just turns off MyConstraint, but retains the position and the configuration associated with MyConstraint.
219  Now, if we want to re-enable the constraint, its just another one-liner:
220  <div style="background-color: #cccccc">
221  <blockquote><pre>
222  Constraints.enable(desc, MyConstraint.class);
223  </pre></blockquote></div>
224  <p>
225  Similarly, constraints on the entire table are disabled via:
226  <div style="background-color: #cccccc">
227  <blockquote><pre>
228  Constraints.disable(desc);
229  </pre></blockquote></div>
230  <p>
231  Or enabled via:
232 
233  <div style="background-color: #cccccc">
234  <blockquote><pre>
235  Constraints.enable(desc);
236  </pre></blockquote></div>
237  <p>
238  Lastly, suppose you want to remove MyConstraint from the table, including with position it should be run at and its configuration.
239  This is similarly simple:
240  <div style="background-color: #cccccc">
241  <blockquote><pre>
242  Constraints.remove(desc, MyConstraint.class);
243  </pre></blockquote></div>
244  <p>
245  Also, removing <i>all</i> constraints from a table is similarly simple:
246  <div style="background-color: #cccccc">
247  <blockquote><pre>
248  Constraints.remove(desc);
249  </pre></blockquote></div>
250  This will remove all constraints (and associated information) from the table and turn off the constraint processing.
251  <p><b>NOTE</b><p>
252  It is important to note the use above of
253  <div style="background-color: #cccccc">
254  <blockquote><pre>
255  Configuration conf = new Configuration(false);
256  </pre></blockquote></div>
257  If you just use <code> new Configuration()</code>, then the Configuration will be loaded with the default
258  properties. While in the simple case, this is not going to be an issue, it will cause pain down the road.
259  First, these extra properties are going to cause serious bloat in your {@link org.apache.hadoop.hbase.HTableDescriptor},
260  meaning you are keeping around a ton of redundant information. Second, it is going to make examining
261  your table in the shell, via <code>describe 'table'</code>, a huge pain as you will have to dig through
262  a ton of irrelevant config values to find the ones you set. In short, just do it the right way.
263  */
